Definición e Importancia

¿Qué son los Requerimientos?

Los requerimientos son descripciones de las necesidades o deseos de un producto. Son la base para todo el trabajo de desarrollo posterior y describen no solo lo que el software debe hacer sino también las restricciones sobre su operación e implementación.

Características de un buen requerimiento

  • 1 Completo: Define todo lo necesario
  • 2 Correcto: Representa fielmente las necesidades
  • 3 Factible: Puede implementarse con los recursos disponibles
  • 4 Necesario: Su ausencia causaría una deficiencia en el sistema
  • 5 Priorizable: Puede clasificarse según su importancia
  • 6 Verificable: Puede comprobarse que se ha implementado correctamente

Importancia en el Ciclo de Desarrollo

Una especificación adecuada de requerimientos es fundamental para el éxito de un proyecto de software. Los estudios demuestran que los errores en los requerimientos son:

Herramienta ES.RE (Norma IEEE Std. 830-1998)

El estándar IEEE 830-1998 proporciona directrices para la especificación de requisitos de software (SRS) y establece la estructura y contenido que deben tener estos documentos.

Estructura del Documento SRS

1

Introducción

  • Propósito del documento
  • Alcance del sistema
  • Definiciones y acrónimos
  • Referencias
  • Visión general del documento
2

Descripción General

  • Perspectiva del producto
  • Funciones del producto
  • Características de los usuarios
  • Restricciones
  • Suposiciones y dependencias
3

Requisitos Específicos

  • Interfaces externas
  • Requisitos funcionales
  • Requisitos no funcionales
  • Restricciones de diseño
  • Atributos del sistema
4

Información de Apoyo

  • Apéndices
  • Índices

Beneficios de Utilizar IEEE 830

Documentación Estructurada

Proporciona una estructura clara y organizada para capturar todos los requisitos relevantes.

Reutilización

Facilita la reutilización de especificaciones en proyectos similares o relacionados.

Calidad Garantizada

Asegura que se consideren todos los aspectos importantes de los requerimientos.

Comunicación Efectiva

Proporciona un lenguaje común entre clientes, usuarios y equipo de desarrollo.

Consideraciones

Aunque IEEE 830 proporciona una base sólida, es importante adaptarla a las necesidades específicas del proyecto y complementarla con otras técnicas modernas de ingeniería de requisitos, especialmente en entornos ágiles.

Tipos de Requerimientos

Requerimientos Funcionales

Describen los servicios que el sistema debe proporcionar, cómo debe reaccionar a entradas particulares y cómo debe comportarse en situaciones específicas.

Ejemplos de Requerimientos Funcionales

RF01
El sistema debe permitir a los usuarios registrados iniciar sesión mediante email y contraseña.
RF02
El sistema deberá enviar notificaciones por email cuando una nueva tarea sea asignada a un usuario.
RF03
El sistema debe permitir la generación de reportes de ventas mensuales en formatos PDF y Excel.

Características

  • Describen el qué debe hacer el sistema
  • Definidos desde la perspectiva del usuario
  • Expresados como capacidades específicas
  • Deben ser verificables y trazables

Requerimientos No Funcionales

Especifican criterios para evaluar la operación de un sistema, en lugar de comportamientos específicos. Definen restricciones o atributos de calidad.

Rendimiento

Ejemplo: El sistema debe responder a las consultas en menos de 2 segundos para el 90% de las transacciones.

Seguridad

Ejemplo: El sistema debe cifrar todas las contraseñas de usuarios utilizando el algoritmo SHA-256.

Disponibilidad

Ejemplo: El sistema debe estar disponible el 99.9% del tiempo durante horario comercial.

Usabilidad

Ejemplo: El sistema debe permitir a un usuario nuevo completar un proceso de compra en menos de 5 pasos.

Escalabilidad

Ejemplo: El sistema debe ser capaz de manejar un aumento del 200% en el número de usuarios sin degradación del rendimiento.

Usuarios del Sistema

Tipos de Usuarios y Clasificación

Usuarios Primarios

Interactúan directamente con el sistema de forma habitual y dependen de él para realizar su trabajo.

Operadores Clientes regulares Administradores

Usuarios Secundarios

Usan el sistema ocasionalmente o indirectamente para obtener informes o resultados.

Gerentes Auditores Usuarios ocasionales

Usuarios Terciarios

Importancia de la Clasificación

Clasificar adecuadamente a los usuarios permite establecer prioridades en los requisitos, diseñar interfaces apropiadas para cada perfil y garantizar que se abordan las necesidades de todos los interesados.

Técnicas de Relevamiento

Cuestionarios

Los cuestionarios son herramientas estructuradas que permiten recopilar información de múltiples personas de manera eficiente. Pueden ser útiles para obtener datos cuantitativos y cualitativos sobre necesidades y preferencias.

Ventajas

  • Permiten recopilar datos de muchos usuarios rápidamente
  • Proveen resultados cuantificables
  • No requieren la presencia del analista
  • Pueden completarse en el momento conveniente para el encuestado

Desventajas

  • Limitados para capturar información detallada o matices
  • No permiten hacer preguntas de seguimiento inmediatas
  • Pueden tener baja tasa de respuesta
  • Riesgo de malinterpretación de las preguntas

Recomendaciones

  • Mantener cuestionarios concisos (menos de 15 minutos para completar)
  • Incluir tanto preguntas cerradas como abiertas
  • Probar el cuestionario antes de distribuirlo masivamente
  • Explicar claramente el propósito y confidencialidad
  • Considerar incentivos para aumentar la participación

Revisión de Registros

La revisión de registros consiste en analizar documentación existente, como manuales de procedimientos, informes, datos históricos y cualquier documentación relevante para comprender los procesos actuales y las necesidades del sistema.

Tipos de Documentos a Revisar

Manuales de usuario existentes
Informes y estadísticas
Diagramas de flujo de procesos
Formatos y plantillas
Normativas y regulaciones

Beneficios

La revisión de documentación proporciona contexto valioso sobre los procesos existentes, evita reinventar soluciones y ayuda a identificar requisitos que podrían no surgir en entrevistas o talleres.

Entrevistas

Las entrevistas son conversaciones estructuradas o semiestructuradas con stakeholders para obtener información detallada sobre necesidades, expectativas y problemas. Son una de las técnicas más efectivas para la elicitación de requisitos.

Entrevistas Estructuradas

Siguen un guión predefinido de preguntas, generalmente en un orden específico.

Preguntas preparadas Fácil de analizar Menos flexibilidad

Entrevistas Semiestructuradas

Combinan preguntas predefinidas con la posibilidad de explorar temas que surjan durante la conversación.

Guía flexible Exploración de temas Balance de control

Entrevistas No Estructuradas

Conversaciones abiertas donde la discusión fluye naturalmente según los temas que surjan.

Máxima libertad Descubrimiento Difícil de controlar

Proceso de Entrevista

1
Preparación

Investigar al entrevistado y preparar guía de preguntas.

2
Introducción

Presentarse y explicar el propósito de la entrevista.

3
Desarrollo

Realizar preguntas y explorar respuestas con técnicas apropiadas.

4
Cierre

Resumir puntos clave y acordar próximos pasos.

5
Documentación

Transcribir y analizar la información obtenida.

Técnicas Efectivas

  • Escucha activa: Prestar atención completa a lo que dice el entrevistado
  • Preguntas abiertas: Usar preguntas que no se respondan con sí/no
  • Parafraseo: Reformular lo que ha dicho el entrevistado para confirmar entendimiento
  • Exploración: Profundizar en temas relevantes con preguntas de seguimiento
  • Silencio estratégico: Permitir pausas para que el entrevistado reflexione

Factibilidades del Sistema

El estudio de factibilidad es una evaluación que determina si un proyecto es viable desde diferentes perspectivas. Este análisis ayuda a identificar obstáculos potenciales y a tomar decisiones informadas antes de comprometer recursos significativos.

Factibilidad Operativa

Evalúa si el sistema propuesto funcionará adecuadamente en la organización y será utilizado correctamente por los usuarios.

Aspectos a Evaluar

Aceptación de Usuarios

¿Los usuarios están dispuestos a adoptar el nuevo sistema?

Capacitación

¿Existen recursos para capacitar adecuadamente al personal?

Cambios en Procesos

¿Los cambios en flujos de trabajo son aceptables?

Soporte Organizacional

¿La dirección apoya el proyecto y sus implicaciones?

Herramientas de Evaluación

  • Entrevistas con usuarios y gerentes
  • Encuestas de actitud y disposición
  • Evaluación de la resistencia al cambio
  • Análisis de impacto en procesos existentes

Factibilidad Técnica

Evalúa si la organización tiene o puede adquirir la tecnología necesaria para implementar y operar el sistema propuesto.

Hardware

  • Disponibilidad de equipos necesarios
  • Capacidad de procesamiento requerida
  • Infraestructura de red y comunicaciones
  • Almacenamiento y servidores

Software

  • Tecnologías y plataformas requeridas
  • Disponibilidad de expertise en desarrollo
  • Compatibilidad con sistemas existentes
  • Escalabilidad de la solución

Competencias

  • Habilidades técnicas disponibles
  • Necesidades de contratación o capacitación
  • Soporte técnico y mantenimiento

Lógica del Sistema

Árbol de Decisión

Un árbol de decisión es una representación gráfica de un proceso de toma de decisiones, donde cada nodo representa una condición y cada rama representa un posible resultado o acción.

Aplicaciones en Ingeniería de Requisitos

  • Modelar flujos de decisión complejos
  • Visualizar reglas de negocio
  • Ilustrar escenarios condicionales
  • Documentar lógica de validación
  • Representar algoritmos de toma de decisiones

Ventajas

Los árboles de decisión son herramientas visuales poderosas para comunicar procesos complejos de forma clara y comprensible para stakeholders técnicos y no técnicos.

Tabla de Decisión

Una tabla de decisión es una herramienta que organiza condiciones y acciones en formato tabular, permitiendo visualizar todas las combinaciones posibles de condiciones y sus correspondientes resultados.

Cuándo usar cada herramienta

Tablas de decisión: Ideal para situaciones con muchas combinaciones de condiciones y cuando es importante verificar la exhaustividad.

Árboles de decisión: Mejor para ilustrar procesos de decisión secuenciales y cuando la visualización jerárquica facilita la comprensión.
