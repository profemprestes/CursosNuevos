Modelo Esencial

El modelo esencial es una descripción del sistema que muestra qué debe hacer el sistema sin mostrar cómo se implementará. Se centra en la esencia del problema, independiente de la tecnología, y está compuesto principalmente por el modelo ambiental y el modelo de comportamiento.

Modelo Ambiental

El modelo ambiental define el entorno en el que opera el sistema. Identifica las entidades externas con las que el sistema interactúa y define los límites o fronteras del sistema.

Diagrama de Contexto

Muestra al sistema como una única caja negra en su entorno, conectada a entidades externas a través de flujos de datos.

Lista de Eventos

Identifica todos los eventos externos a los que el sistema debe responder, provenientes de entidades externas o de condiciones temporales.

Ejemplo de Diagrama de Contexto

Modelo de Comportamiento

El modelo de comportamiento describe lo que hace el sistema internamente para responder a los eventos del entorno. Muestra cómo se transforman los datos y cómo se controla esta transformación.

Diagramas de Flujo de Datos (DFD)

Muestran cómo los datos se mueven a través del sistema, ilustrando los procesos, almacenes de datos y flujos de información.

Máquinas de Estado

Describen los estados en los que puede encontrarse el sistema y cómo cambia entre ellos en respuesta a eventos.

Aspectos Clave

  • El modelo esencial representa lo que el sistema debe hacer, no cómo debe implementarse
  • Se desarrolla en las etapas iniciales del análisis de requisitos
  • Es independiente de la tecnología y plataforma
  • Ayuda a los stakeholders a comprender la funcionalidad del sistema

Lista de Acontecimientos

La lista de acontecimientos identifica todos los eventos a los que el sistema debe responder. Es un componente crucial del modelo esencial que ayuda a estructurar el análisis del sistema.

Evento Origen Respuesta del Sistema Destino
Solicitud de matrícula Estudiante Registro de matrícula, verificación de disponibilidad Base de datos, Administración
Registro de calificaciones Profesor Almacenamiento de calificaciones, cálculo de promedios Base de datos, Estudiante
Pago de cuota Sistema de Pagos Actualización de estado financiero, emisión de recibo Estudiante, Administración
Fin de período académico Tiempo Generación de reportes estadísticos, cierre de cursos Administración

Beneficios de la Lista de Acontecimientos

Enfoca el análisis en eventos específicos que requieren respuesta
Ayuda a establecer los límites del sistema
Facilita la identificación de casos de uso
Proporciona una base para la validación de requisitos

UML (Lenguaje Unificado de Modelado)

Introducción a UML

El Lenguaje Unificado de Modelado (UML) es un lenguaje visual estándar para modelar software. Proporciona una manera de visualizar, especificar, construir y documentar artefactos de un sistema.

Origen e Historia

UML surgió en 1997 como una unificación de varios métodos de modelado orientado a objetos. Fue desarrollado por Grady Booch, James Rumbaugh y Ivar Jacobson (conocidos como "los tres amigos") y adoptado por el Object Management Group (OMG) como estándar.

Conceptos Fundamentales

Notación Estándar

Símbolos y reglas consistentes para representar elementos del sistema

Diferentes Vistas

Múltiples perspectivas del sistema (estructural, comportamiento, interacción)

Orientado a Objetos

Basado en los principios de la programación orientada a objetos

Diagrama de Navegabilidad

Los diagramas de navegabilidad muestran la estructura de navegación de una aplicación, especialmente útiles para aplicaciones web y móviles. Representan cómo los usuarios pueden moverse entre las diferentes pantallas o páginas del sistema.

Elementos Básicos

Nodos

Representan pantallas, páginas o vistas de la aplicación

Enlaces

Muestran las transiciones o navegación posible entre nodos

Puntos de Inicio

Indican las pantallas iniciales del sistema

Notación Común

Aunque no forma parte del estándar UML oficial, los diagramas de navegabilidad suelen utilizar las siguientes convenciones:

  • Rectángulos para representar páginas o pantallas
  • Flechas para mostrar la dirección de navegación
  • Etiquetas en las flechas para describir la acción que provoca la navegación
  • Colores o formas diferentes para categorizar tipos de pantallas

Ejemplo de Diagrama de Navegabilidad

Usos y Beneficios

  • Facilita la planificación de la arquitectura de información
  • Ayuda a identificar posibles puntos muertos o callejones sin salida
  • Proporciona una visión clara del flujo de usuario
  • Sirve como referencia para diseñadores de UX/UI
  • Complementa otros diagramas UML durante el desarrollo

Diagramas de Casos de Uso

Fundamentos

Los diagramas de casos de uso modelan la funcionalidad del sistema tal como la ven los actores externos. Representan un conjunto de acciones (casos de uso) que el sistema puede realizar en colaboración con uno o más usuarios externos (actores).

Elementos Básicos

Actor

Representa a cualquier entidad externa (persona, sistema, dispositivo) que interactúa con el sistema. Se representa con una figura de "stick man".

Caso de Uso

Representa una funcionalidad o servicio que el sistema proporciona a los actores. Se representa con una elipse con el nombre dentro o debajo.

Límite del Sistema

Representa los límites del sistema que contiene los casos de uso. Se dibuja como un rectángulo que rodea los casos de uso.

Relaciones entre Casos de Uso

Asociación
Comunicación entre un actor y un caso de uso
Include
Un caso de uso incorpora el comportamiento de otro
Extend
Un caso de uso extiende otro en situaciones específicas
Generalización
Un caso de uso es una especialización de otro

Ejemplo de Diagrama de Casos de Uso

Documentación de Casos de Uso

Los diagramas de casos de uso suelen complementarse con descripciones textuales detalladas que incluyen:

Nombre Solicitar Préstamo

Diagramas de Clases

Los diagramas de clases son el pilar de los modelos orientados a objetos. Muestran la estructura estática del sistema, representando las clases, sus atributos, operaciones y las relaciones entre ellas. Son fundamentales para el diseño de sistemas orientados a objetos.

Estructura de una Clase

Una clase en UML se representa como un rectángulo dividido en tres compartimentos:

1. Nombre de la Clase

Identificador único de la clase, generalmente un sustantivo.

2. Atributos

Propiedades o características de la clase, con su tipo de dato.

visibilidad nombre: tipo [= valor inicial]

3. Operaciones

Métodos o funciones que puede realizar la clase.

visibilidad nombre(parámetros): tipo_retorno

Visibilidad

Símbolo Visibilidad Descripción
+ public Accesible por cualquier clase
- private Accesible solo dentro de la clase
# protected Accesible por la clase y sus subclases
~ package Accesible dentro del mismo paquete

Relaciones entre Clases

Asociación

Conexión semántica entre clases. Se representa con una línea sólida.

Agregación

Relación "parte-de" débil. Se representa con un diamante vacío.

Composición

Relación "parte-de" fuerte. Se representa con un diamante relleno.

Herencia

Relación "es-un". Se representa con una flecha triangular vacía.

Multiplicidad

Indica cuántas instancias de una clase pueden relacionarse con una instancia de la otra clase.

Notación Significado
1 Exactamente uno
0..1 Cero o uno
* Cero o más
1..* Uno o más
n..m Desde n hasta m

Diagramas de Secuencia

Propósito y Elementos

Los diagramas de secuencia ilustran la interacción entre objetos en una secuencia temporal. Muestran cómo los procesos operan entre sí y en qué orden, enfocándose en el intercambio de mensajes entre objetos.

Línea de Vida

Representa un participante individual en la interacción. Se dibuja como una línea vertical punteada desde un rectángulo con el nombre del participante.

Mensaje

Comunicación entre participantes. Se representa como una flecha entre líneas de vida con el nombre del mensaje.

Activación

Período durante el cual un participante está ejecutando una operación. Se representa como un rectángulo delgado sobre la línea de vida.

Tipos de Mensajes

→
Síncrono: El remitente espera una respuesta antes de continuar
⟶
Asíncrono: El remitente continúa sin esperar respuesta
⟵⟵
Respuesta: Retorno de un mensaje síncrono
⟶×
Creación: Crea una nueva instancia del objeto destino

Ejemplo de Diagrama de Secuencia

Cuándo Usar Diagramas de Secuencia

  • Para modelar escenarios de casos de uso
  • Para visualizar la lógica de métodos complejos
  • Para entender el flujo de mensajes en sistemas distribuidos
  • Para identificar responsabilidades de objetos
  • Para detectar cuellos de botella en la comunicación

Diagramas de Actividad

Fundamentos

Los diagramas de actividad representan flujos de trabajo paso a paso de operaciones componentes en un sistema. Son esencialmente diagramas de flujo sofisticados que muestran el flujo de control de una actividad a otra.

Nodo Inicial

Punto de inicio del flujo. Se representa como un círculo negro sólido.

Nodo Final

Punto de terminación del flujo. Se representa como un círculo con un punto negro dentro.

Actividad

Paso en el flujo de trabajo. Se representa como un rectángulo con esquinas redondeadas.

Decisión

Punto de bifurcación condicional. Se representa como un rombo.

Barra de Sincronización

Divide o une flujos concurrentes. Se representa como una línea horizontal o vertical gruesa.

Ejemplo de Diagrama de Actividad

Aplicaciones de los Diagramas de Actividad

  • Modelar flujos de trabajo de negocio
  • Describir procedimientos complejos
  • Ilustrar la lógica de algoritmos
  • Modelar operaciones paralelas en sistemas concurrentes
  • Documentar procesos de negocio y casos de uso

Patrones Comunes en Diagramas de Actividad

Decisión-Fusión

Representa bifurcaciones condicionales en el flujo con un rombo para la decisión y otro para reunir los caminos alternativos.

Fork-Join

Muestra actividades que pueden ejecutarse en paralelo con una barra de sincronización que divide el flujo y otra que lo reúne.

Bucle

Representa flujos que se repiten hasta que se cumple una condición, usando decisiones para controlar la iteración.
